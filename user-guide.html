Overview

Aquarium Controller is a Node.js web application that runs on your local network and connects to Home Assistant via its WebSocket API. It provides a mobile-optimised interface for managing your aquarium's lighting, circulation pumps, dosing pumps, sensors, and power switches.

The app does not require any cloud services — everything runs locally. Home Assistant is used as the bridge to your physical devices (ESPs, smart plugs, etc.), and the app talks to HA directly over your LAN.

What the App Controls Directly

Some features are handled entirely by the Node.js server, without needing HA automations:

FeatureHow it worksUpdate interval
Lighting scheduleServer loop calls light.turn_on with computed brightnessEvery 60 seconds
Circulation pump modesServer loop calls fan.set_percentage based on mode algorithmEvery 2 seconds
Switch schedules & sensor automationsServer loop checks time/sensor values and toggles switchesEvery 30 seconds
Dosing scheduleServer loop fires input_boolean trigger at configured intervalsEvery 30 seconds
ℹ️
Important Because the lighting, pump, and dosing loops run inside the Node.js process, the server must be running continuously for schedules to be enforced. Consider running it as a system service (e.g. pm2 or systemd).

Prerequisites

RequirementNotes
Node.js 18+Run node -v to check. Download from nodejs.org if needed.
Home AssistantAny recent version. Must be reachable on your local network.
HA Long-Lived Access TokenGenerated in your HA profile — see Section 3.
Devices in HAYour lights, pumps, and switches must already be added to HA as entities.

Starting the Server

Place all project files in a folder (e.g. ~/aquarium).
Run npm install in that folder to install dependencies (express and ws).
Run node server.js to start the server. By default it listens on port 4000.
Open http://<your-server-ip>:4000 in a browser on your phone or computer.
💡
Tip — Run on startup Install pm2 globally (npm install -g pm2), then run pm2 start server.js --name aquarium followed by pm2 save and pm2 startup to keep it running across reboots.

Connecting to Home Assistant

Generating a Long-Lived Access Token

The app authenticates with HA using a Long-Lived Access Token (LLAT). This is a personal token tied to your HA user account.

In Home Assistant, click your username / profile picture in the bottom-left sidebar.
Scroll to the Security section at the bottom of the profile page.
Under Long-Lived Access Tokens, click Create Token.
Give it a name (e.g. Aquarium Controller) and click OK.
Copy the token immediately — it will not be shown again. Paste it somewhere safe temporarily.
⚠️
Keep your token secure The token grants full access to your HA instance. Do not share it or commit it to version control. It is stored in config.json on your server.

Entering Connection Details

Open the app and tap the ⚙ gear icon in the top-right corner to open Settings.
Tap the Connection card under Home Assistant to expand it.
Enter your HA Host / IP Address (e.g. 192.168.1.123). Do not include http://.
Leave the Port as 8123 unless you have changed it in HA.
Paste your token into the Long-Lived Access Token field.
Optionally tap Test to verify the connection before saving.
Tap Save & Connect. The badge will turn green when connected.

Once connected, all entity dropdown selectors throughout Settings will be populated from your live HA entity list. Use the ↻ Refresh button at the top of the Lighting Channels section to reload the entity list at any time.

Setting Up HA Helpers (Dosing)

The dosing system uses three HA helpers per pump to store calibration and dosing data in HA itself, keeping it in sync even if the app is restarted. You need to create these helpers manually in HA before configuring dosing in the app.

ℹ️
Where to create helpers In HA go to Settings → Devices & Services → Helpers and click + Create Helper for each one below. Repeat for each dosing pump.

input_number Helpers

Create two input_number helpers per pump:

input_number
Calibration Multiplier Stores seconds-per-mL for this pump. Suggested name: calibration_multiplier_1 (giving entity ID input_number.calibration_multiplier_1). Set the range to 0.01 – 100, step 0.0001. Initial value: 1.
input_number
Dose Volume Stores the target dose in mL. Suggested name: dosing_volume_1 (entity ID input_number.dosing_volume_1). Set range to 0.1 – 100, step 0.1. Initial value: your desired dose (e.g. 2).

input_boolean Trigger

Create one input_boolean helper per pump:

input_boolean
Dose Trigger Acts as a signal to start the dosing automation. Suggested name: dose_trigger_pump_1 (entity ID input_boolean.dose_trigger_pump_1). The app turns this ON to request a dose; your HA automation handles the rest and turns it back OFF.

Dosing Automation

Create one automation per pump that responds to the trigger boolean. The automation should:

  1. Trigger when input_boolean.dose_trigger_pump_1 turns ON
  2. Turn ON the pump switch entity (e.g. switch.dosing_pump_1)
  3. Wait for input_number.dosing_volume_1 × input_number.calibration_multiplier_1 seconds
  4. Turn OFF the pump switch entity
  5. Turn OFF input_boolean.dose_trigger_pump_1

Here is the equivalent YAML for the HA automation editor:

alias: Dose Pump 1
trigger:
  - platform: state
    entity_id: input_boolean.dose_trigger_pump_1
    to: "on"
action:
  - service: switch.turn_on
    target:
      entity_id: switch.dosing_pump_1
  - delay:
      seconds: >
        {{ states('input_number.dosing_volume_1') | float *
           states('input_number.calibration_multiplier_1') | float }}
  - service: switch.turn_off
    target:
      entity_id: switch.dosing_pump_1
  - service: input_boolean.turn_off
    target:
      entity_id: input_boolean.dose_trigger_pump_1
mode: single
💡
Naming convention Keep the naming consistent — replace the trailing _1 with _2, _3, etc. for each additional pump. This makes it easy to identify which helpers belong to which pump.

Configuring the App

All configuration is done in the Settings page (⚙ icon). Each section is a collapsible card. Tap the card header to expand it. Changes within each section are saved independently with their own Save button — expanding a section does not save it automatically.

Lighting Channels

Add one entry per LED channel. Each channel maps a friendly name to a HA light entity.

FieldRequiredDescription
NameRequiredDisplay name shown in the app (e.g. Blue, Royal Blue, Cool White)
Light EntityRequiredThe HA light entity for this channel. Must support brightness (0–255).
ℹ️
Light entity requirements Each light entity must support the brightness attribute. PWM-controlled LEDs via an ESP32 running ESPHome work well. RGBW or CCT lights may need a template entity to expose a single brightness channel.

Dosing Pumps

Add one entry per dosing pump. Each pump requires up to four HA entities:

FieldRequiredDescription
NameRequiredDisplay name (e.g. Calcium, Alkalinity, Magnesium)
Switch EntityRequiredThe switch entity that physically controls the pump. Used for the manual ON/OFF priming toggle.
Calibration MultiplierRequiredThe input_number helper storing seconds-per-mL (see Section 4).
Dose VolumeRequiredThe input_number helper storing the target dose in mL (see Section 4).
Trigger BooleanRequiredThe input_boolean helper that fires the dosing automation (see Section 4).

Tap Save Dosing after configuring pumps. Dosing schedules are configured on the Dosing page itself, not here.

Circulation Pumps

Add one entry per circulation pump.

FieldRequiredDescription
NameRequiredDisplay name (e.g. Return Pump, Wavemaker 1)
Switch / Fan EntityRequiredThe HA entity for this pump. Must be a fan entity with set_percentage support for variable speed modes. A plain switch entity works too but only supports Constant mode at 100%.
⚠️
Fan entities for variable speed For Wave, Pulse, and Random modes to work, the pump must be exposed to HA as a fan entity that supports the fan.set_percentage service. Many ESPHome-based controllers and Shelly Dimmer devices support this. Check your entity in HA's Developer Tools → States to confirm the entity domain is fan and it has a percentage attribute.

Switches

Add one entry per controllable switch (e.g. refugium light, UV steriliser, heater).

FieldRequiredDescription
Display NameRequiredFriendly name shown in the app
Switch EntityRequiredAny HA switch entity
Rated WattsOptionalUsed in the power summary display
IconOptionalAny emoji character. Defaults to ⏻

Sensors

The sensors section has two parts: the four built-in sensor groups, and custom sensors.

Built-in Sensors

Four groups are pre-configured with display ranges and optimal zones: Temperature, pH, ORP, and KH/Alkalinity. For each one you can:

  • Set a Primary sensor entity (the main value displayed)
  • Set a Secondary / Backup sensor entity (shown in the dual-probe comparison)
  • Toggle Active to show or hide the sensor from the dashboard and sensors page

Custom Sensors

Tap + Add Custom Sensor to add any additional sensor entity. Each custom sensor has:

FieldDescription
NameDisplay name (e.g. RO Level, TDS, Salinity)
Sensor EntityAny HA sensor entity
UnitChoose from: °C, °F, ppm, %, ‰, mV, dKH, none, or custom (enter your own)
Show on HomeIf enabled, this sensor appears as a tile on the main dashboard
Show GraphIf enabled, this sensor appears on the Sensors page with an 8h history chart

Using the Lighting Page

The lighting page lets you set a daily schedule for each channel, controlling when they turn on and off and how brightness ramps up and down during sunrise/sunset periods.

Schedule Settings (per channel)

SettingDescription
Start TimeWhen the channel begins ramping up from 0%
End TimeWhen the channel begins ramping down to 0% (turns off at this time)
Ramp Period (hours)Duration of the sunrise and sunset ramp. E.g. 3 hours means the channel spends 3 hours ramping up and 3 hours ramping down. Must be less than half the total on-window.
Max BrightnessPeak brightness value (0–255) during the full-intensity midday period
Graph ColourThe colour used for this channel's line on the schedule graph

The page shows a live preview graph of all channels over a 24-hour period so you can visualise the light cycle before saving.

💡
Staggered channels For a natural look, stagger your channels slightly. For example, set Blue to start at 08:00 and white channels to start at 08:30–09:00, with the whites having a lower max brightness. This mimics the way natural light shifts colour temperature through the day.

Tap Save Schedule to apply. The server loop applies brightness changes every 60 seconds based on Sydney local time — the first update after saving may take up to a minute.

Using the Pumps Page

Each circulation pump has an on/off toggle and a live speed bar, followed by a mode selector and mode-specific settings.

Flow Modes

ModeSettingsBehaviour
〰️ Constant Speed % Runs at a fixed speed continuously. The server sets the fan percentage once and only updates when the setting changes.
🌊 Wave Min %, Max %, Period (seconds) Speed follows a smooth sine wave between min and max over the configured period. Updated every 2 seconds for a fluid motion effect.
💓 Pulse Min %, Max %, Pulse Length (0.5–3s), Delay (seconds) Alternates sharply between Max for the pulse duration and Min for the delay duration. Good for powerheads that simulate surge/tide.
🎲 Random Min %, Max %, Change Rate (Slow/Medium/Fast) Random walk between min and max. Speed changes by ±2% (slow), ±5% (medium), or ±10% (fast) every 2 seconds. Mimics natural turbulence.
ℹ️
Variable speed requirement Wave, Pulse, and Random modes require the pump entity to be a fan domain entity with set_percentage support in HA. If your pump is a simple on/off switch, only Constant mode at 100% is meaningful.

Select a mode, adjust the settings, then tap 💾 Save. The server loop immediately begins using the new settings. The live speed bar at the top of each card reflects the actual percentage currently being sent to HA.

Using the Dosing Page

Manual Control

Each pump card has two control buttons:

ON/OFF
Toggle (Priming) Turns the pump switch directly on or off. Use this for priming lines after initial setup or when replacing tubing. The pump will run continuously until you turn it off.
DOSE
Run One Dose Fires the input_boolean trigger, which starts your HA automation. The automation runs the pump for exactly (dose volume × calibration multiplier) seconds, then turns it off. The button is disabled if no trigger boolean is configured.

Dose Volume

The current dose volume (read live from the HA input_number helper) is shown on each card. To change it, enter a new value in the input field and tap Save — this writes directly back to the HA helper, keeping HA and the app in sync.

Dosing Schedule

Each pump has its own schedule, configured directly on this page:

SettingDescription
Schedule EnabledToggle to activate or pause automatic dosing for this pump without losing settings
All HoursIf ON, the pump doses at the configured interval around the clock. If OFF, dosing is restricted to between Start Time and End Time.
Start / End TimeActive window for dosing (only visible when All Hours is OFF)
IntervalHow often to dose, in minutes or hours
Offset from HourShifts the dose time within each hour. E.g. if Calcium has offset 15 and Alkalinity has offset 45, they dose 30 minutes apart every hour — useful to avoid precipitation when dosing competing supplements.

Tap 💾 Save Schedule on each pump after changing schedule settings.

⚠️
Never dose calcium and alkalinity simultaneously Always use the offset setting to ensure at least 15–30 minutes between calcium and alkalinity doses. Dosing them at the same time causes precipitation in the water column.

Calibration

Calibration establishes how many seconds the pump needs to run to deliver 1 mL. The result is stored as a multiplier (sec/mL) in the HA input_number helper.

Select the pump to calibrate from the dropdown.
Prepare a measuring vessel under the pump outlet.
Tap ▶ Dose 10 mL. The pump will run for 10 × its current multiplier seconds.
Measure the actual output in mL.
Enter the measured value and tap 💾 Save Calibration.

The new multiplier is calculated as: new multiplier = (10 ÷ measured mL) × current multiplier. This proportional correction means you can calibrate iteratively — each calibration brings the pump closer to exact. After two or three calibrations the accuracy will be very high.

💡
First calibration If the pump has never been calibrated, the multiplier will be 1 (1 second per mL). Run the Dose 10 mL function, measure the output, and save. The new multiplier will be set accordingly. Repeat once more for improved accuracy.

Using the Sensors Page

The sensors page shows all active sensors with live readings and 8-hour history charts.

Built-in Sensors

Temperature, pH, ORP, and KH each display:

  • A large current reading
  • A status bar showing where the value falls within the absolute range, with markers for the optimal zone
  • A coloured pill indicating ✓ Optimal, ⚠ Caution, or ⚠ Out of Range
  • An 8-hour sparkline chart with min/max annotation
  • A dual-probe comparison if a secondary sensor is configured
SensorOptimal rangeAbsolute range
Temperature24–27 °C22–30 °C
pH8.1–8.47.8–8.6
ORP300–450 mV200–500 mV
KH8–12 dKH6–14 dKH

Custom Sensors

Custom sensors configured in Settings with Show Graph enabled appear at the bottom of the sensors page with a live value and sparkline chart. They do not have optimal/absolute range indicators.

ℹ️
History data The server records sensor history in memory and persists it to history.json every 5 minutes. History is retained for up to 24 hours (288 readings at 5-minute intervals). If the server is restarted, up to 5 minutes of recent readings may be lost, but older data will be restored from disk.

Using the Switches Page

Each switch can be controlled manually or automated by the server. Tap the toggle on any switch card to turn it on or off immediately.

Automation Modes

Each switch has an expandable automation panel with three modes:

ModeBehaviour
ManualNo automation. The switch only responds to manual taps in the app or changes made in HA.
ScheduleThe server turns the switch on at a configured time and off at another. Checked every 30 seconds. Good for refugium lighting, UV sterilisers, etc.
SensorThe switch toggles based on a sensor value crossing a threshold. Configure a sensor entity, trigger type (high or low), setpoint, and variance band. For example, a heater could be set to turn on when temperature falls below 25.5°C and off when it reaches 26.5°C.
💡
Sensor-based automation with variance The variance setting creates a hysteresis band to prevent rapid on/off cycling. A setpoint of 26°C with variance of 1°C means the switch turns on at 25.5°C and off at 26.5°C (for "low" trigger type).

How It All Works

Understanding the architecture helps when things don't behave as expected.

Communication Flow

Browser (phone/computer)
        ↕  HTTP  (port 4000)
Node.js server  (server.js)
        ↕  WebSocket  (HA WebSocket API, port 8123)
Home Assistant
        ↕  Zigbee / Z-Wave / Wi-Fi / ESPHome / etc.
Physical devices (lights, pumps, sensors)

The Node.js server maintains a persistent WebSocket connection to HA. It subscribes to all state change events, so its internal state cache is always up to date. When the app sends a command (e.g. turn on a pump), the server calls the appropriate HA service via the WebSocket. HA then relays that to the physical device.

Server-Side Loops

LoopIntervalWhat it does
Lighting60 sComputes brightness for each channel using a trapezoidal curve (ramp up → hold → ramp down) and calls light.turn_on or light.turn_off
Pump modes2 sComputes target speed for each pump (wave/pulse/random maths) and calls fan.set_percentage. Only sends if value changed.
Switch automation30 sChecks scheduled times and sensor thresholds, toggles switches as needed
Dosing schedule30 sChecks whether any pump is due for a dose based on interval and offset, fires input_boolean.turn_on
History persist5 minWrites sensor history cache to history.json

Data Storage

FileContents
config.jsonHA connection details (host, port, token)
settings.jsonAll app configuration: lighting channels & schedules, pump entities & mode configs, dosing pump entities & schedules, switches, sensors
history.jsonSensor reading history (up to 24h per tracked entity)
⚠️
Back up your settings settings.json contains all your configuration. Keep a backup copy, especially after a complete setup. If the file is lost, you will need to reconfigure everything from scratch.

Troubleshooting

Connection Issues

SymptomLikely cause & fix
"Not configured" in footerNo HA credentials saved. Go to Settings → Connection and enter your host and token.
"Disconnected — retrying…"Server can reach the configured host but the WebSocket failed. Check that HA is running and reachable at the configured IP. The server retries automatically every 10 seconds.
Entity dropdowns show "Connect to HA to see entities"HA is not connected. Resolve the connection issue first, then use the ↻ Refresh button.
Test passes but Save & Connect shows disconnectedThe token is valid but the WebSocket may be taking a moment to authenticate. Wait 5–10 seconds and check the badge again.

Lighting Not Changing

SymptomFix
Lights don't change at scheduled timesThe server loop runs every 60 seconds. Wait up to a minute after saving. Check the server console — it logs every light loop action.
Brightness is always 0Check that Start Time is before End Time and that the current time falls within the window. Check that Max Brightness is set above 0.
Ramp doesn't look rightThe ramp period must be less than half the total on-window (end − start). If it's too long, the ramps overlap and the channel never reaches max brightness.

Dosing Issues

SymptomFix
"Run One Dose" button is greyed outNo trigger boolean is configured for this pump in Settings → Dosing Pumps.
Dose triggers but pump doesn't runCheck the HA automation is enabled and that the trigger entity matches the one configured in the app. Check the pump switch entity is correct.
Dose volume shows "—"The input_number dose volume helper is not configured, or its value in HA is 0. Enter a volume and tap Save.
Calibration multiplier shows 0The calibration helper is not configured or has never been set. Do an initial calibration using the Calibration section on the Dosing page.
Scheduled dose not firingCheck that Schedule Enabled is ON for the pump. Check the server has been restarted after saving the schedule — the dosing loop reads settings on every tick but the schedule state only loads at save time.

Pump Modes Not Working

SymptomFix
Speed stays at a fixed value regardless of modeThe entity is likely a switch not a fan. Check in HA Developer Tools → States that the entity ID starts with fan. and has a percentage attribute.
Live speed bar doesn't moveThe bar reflects what the server is sending. If the pump is OFF (toggle is off), the speed is 0. Turn the pump ON first.

Sensor History Missing

SymptomFix
"Not enough data yet" on sparklineThe server needs at least 2 recorded data points. History is only recorded for entity IDs actively assigned in Settings → Sensors. After assigning a sensor, wait a few minutes for data to accumulate.
Custom sensors not appearing on the pageEnsure Settings → Sensors is saved with both the entity and the "Show Graph" toggle ON. Then restart the server to ensure the new settings are picked up by the history recording loop.

Checking the Server Console

The server logs useful information to the console. If something isn't working as expected, check the terminal where node server.js is running:

  • ✅ HA authenticated — connection is working
  • 📦 Loaded N entities — entity list received from HA
  • LIGHT LOOP → entity | brightness=N — lighting loop is running
  • 💉 Dosing pump N — triggering via … — scheduled dose fired
  • ⏰ Schedule: entity → ON/OFF — switch schedule triggered
  • 🌡️ Sensor: entity → ON/OFF — sensor automation triggered
